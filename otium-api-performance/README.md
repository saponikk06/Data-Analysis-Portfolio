# Технический анализ производительности REST API (Сервис «Otium»)

Данный проект посвящен анализу стабильности и производительности микросервисной архитектуры стримингового сервиса после внедрения новой функциональности фильтрации контента. В основу лег опыт проектирования REST API (JSON, XSD) и агрегации микросервисов.

## 🎯 Бизнес-задача
Оценить влияние изменений в коде на время отклика системы (Latency), выявить потенциальные «узкие места» и проанализировать частоту возникновения ошибок при пиковых нагрузках.

## 🛠 Технологический стек
* **Язык:** Python.
* **Библиотеки:** Pandas, NumPy, Matplotlib, Seaborn.
* **Метрики:** * **Latency:** расчет перцентилей p50, p95, p99.
    * **Error Rate:** процент кодов ответа 4xx и 5xx.
    * **RPS:** интенсивность запросов в секунду.

## 🔍 Этапы анализа
1. **Сбор и предобработка логов:** Анализ данных, содержащих временные метки, эндпоинты (`/get_media`, `/search`, `/user_profile`), время ответа и статус-коды (200, 201, 404, 500).
2. **Расчет перцентилей Latency:** Оценка «длинного хвоста» задержек, которые критичны для пользовательского опыта.
3. **Визуализация распределения:** Построение Boxplot-диаграмм для наглядного сравнения производительности разных методов API.
4. **Анализ ошибок:** Расчет Error Rate в разрезе эндпоинтов для поиска наиболее нестабильных узлов.

## 📈 Ключевые результаты
* **Анализ задержек:** Выявлено, что эндпоинт `/search` имеет наибольшее время отклика в 99-м перцентиле, что подтверждает гипотезу о нагрузке от новых фильтров.
* **Стабильность:** Уровень ошибок (Error Rate) для большинства эндпоинтов остается в пределах нормы (~1.6% - 2.3%), что свидетельствует о корректной агрегации микросервисов.
* **Рекомендации:** Оптимизация логики запросов к БД для эндпоинта поиска и внедрение кеширования результатов фильтрации.
